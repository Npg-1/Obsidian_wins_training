# HTTP #

HTTP(Hyper Text Transfer Protocol): 서버와 클라이언트의 데이터 교환을 **요청(Request)**과 **응답(Response)** 형식으로 정의한 프로토콜
HTTP의 기본 매커니즘은 클라이언트가 서버에게 요청 -> 서버가 응답함, 
웹 서버는 HTTP 서버를 HTTP 서비스 포트에 대기시키고 이 포트는 일반적으로 TCP/80 또는 TCP/8080임

#### *※ 이게 살짝 중요하다고 생각되는데 서버-클라이언트 모델에서 서버는 서비스를 제공하는 입장으로 클라이언트가 서버한테 주문(GET)하고 클라이언트가 서버한테 주문서를 내는(POST) 거여서 어느정도 예외가 있을 수는 있어도 기본적으로는 '클라이언트 -> 서버' 이렇게 동작함* ####


### 네트워크 포트와 서비스 포트 ###
**네트워크 포트** 란 네트워크에 서버와 클라이언트가 정보를 교환하는 추상화된 장소를 의미함. 포트에는 항구라는 의미가 있는데, 클라이언트가 서버의 포트에 접근하여 데이터를 내려놓고, 서버가 클라이언트에 보낼 데이터를 실어서 돌려보내는 장면에서 포트를 떠올리게 됨

**서비스 포트** 란 네트워크 포트 중에서 특정 서비스가 점유하고 있는 포트를 의미함. 예를 들어 HTTP가 80번 포트를 점유하고 있다면 HTTP의 서비스 포트는 80번 포트가 됨

포트로 데이터를 교환하는 방식은 **전송 계층(Transport Layer)** 의 프로토콜을 따름. 대표적으로는 TCP와 UDP가 있음. TCP로 데이터를 전송하려는 서비스에 UDP 클라이언트가 접근하면 데이터 교환이 안 됨. 반대도 똑같음. 그래서 서비스 포트를 표기할 때 전송 프로토콜을 같이 표기하기도함. 예를 들어서 HTTP의 서비스를 80번 포트에서 TCP로 제공하고 있다면 TCP/80 이라고 표기할 수 있음

포트 개수는 일반적으로 2^16개로 0~65535번까지 사용함
포트 번호 중에서 0~1023번 포트는 잘 알려진 포트(Well-Known port)로 각 포트 번호에 유우명한 서비스들이 등록되어 있음
대표적으로 22번에는 SSH, 80에는 HTTP, 443에는 HTTPS가 할당되어 있음

# HTTP 메시지 #

HTTP 메시지에는 클라이언트가 전송하는 HTTP요청, 서버가 반환하는 HTTP 응답이 있음. 요청과 응답은 HTTP 헤더와 바디로 구성되는 공통점이 있음

### HTTP 헤더 ###
HTTP 헤더 각 줄은 CRLF로 구분되며, 첫 줄은 시작 줄(Start Line), 나머지를 헤더라고 부르고 헤더의 마지막은 빈 줄로 나타냄
헤더의 시작줄은 요청과 응답에서 차이가 있지만 이 부분은 뒤에서 기술함
헤더는 필드와 값으로 구성되며, HTTP 메시지 또는 바디의 속성을 나타냄. 하나의 메시지에는 0개 이상의 헤더가 있을 수 있음(없을 수도 있는 지는 모르겠지만 여러 개는 있을 수도 있다)

### HTTP 바디 ###
HTTP 헤더의 끝을 나타내는 CRLF 뒤 부터 메시지의 끝까지 모든 줄을 나타냄. 클라이언트나 서버에게 전송하려는 데이터가 바디에 담김

◇ CRLT ◇
*CRLF는 Carriage Return(CR)과 Line Feed(LF)의 조합을 나타내는 것으로 CR은 커서를 현재 줄의 맨 앞으로 이동시키는 문자이고, LF는 커서를 다음 줄로 이동시키는 문자임. 이것들은 주로 텍스트 파일에서 줄 바꿈을 나타내는데 사용되는 **제어 문자열**로 윈도우에서는 줄을 끝내기 위해 CRLF를 사용하고, 리눅스 같은 유닉스 파생 OS들은 LF를 사용함*

# HTTP 요청 #

HTTP 요청은 클라이언트가 서버에게 특정 동작을 요구하는 메시지임. 서버는 해당 동작이 수행 가능한지, 클라이언트가 동작을 요청할 권한이 있는지 등을 검토하고 적절하면 요청을 처리함

### 시작 줄 ###
HTTP 요청의 시작 줄은 **메서드(Method), 요청 대상(Request target), HTTP 버전**으로 구성됨

**메서드**는 요청 대상에 대해 서버가 수행하길 바라는 동작을 나타냄. HTTP 표준에 정의된 메서드는 8개지만 자주 사용되는 건 GET과 POST가 있음

GET은 리소스를 가져와달라는 요청임. 이용자가 브라우저에 주소를 입력하거나 하이퍼링크를 클릭하면 새로운 페이지를 렌더링 하기 위해서 리소스가 필요한데 이때 브라우저는 GET 요청을 서버에 전송하여 리소스를 받아옴

반대로 POST는 요청 대상에게 데이터를 보내는 메소드로 전송할 데이터는 보통 HTTP 바디에 포함됨. 로그인할 때 입력하는 ID와 비밀번호, 게시판에 작성하는 글 등이 POST로 서버에 보내짐

요청 대상(요청 [[URL, URI|URI]])는 메서드의 대상을 가리키고, HTTP 버전은 클라이언트가 사용하는 HTTP 프로토콜의 버전을 나타냄

# HTTP 응답 #

HTTP 응답은 HTTP 요청에 대한 결과를 반환하는 메시지로 요청을 수행했는지, 하지 않았는지, 안 했다면 왜 안 했는지 같은 상태 정보와 클라이언트에게 전달할 리소스가 포함됨

시작줄
HTTP 응답의 시작 줄은 **HTTP 버전, 상태 코드(Status Code), 처리 사유(Reason Phrase)** 로 구성됨, 각각은 띄어쓰기로 구분됨

 HTTP 버전은 서버에서 사용하는 HTTP 프로토콜의 버전을 나타냄(요청에서는 클라이언트의 버전을 나타내는 것과는 반대임). 상태 코드는 처리 결과를 세 자리수로 나타냄. 처리 사유는 상태 코드가 발생한 이유를 짧게 나타낸 것

# HTTPS #

HTTP 메시지는 평문으로 전달됨. 따라서 중간에 유출되면 중요한 중요가 탈취당할 수 있음
HTTPS(HTTP over Secure socker layer)는 TLS(Transport Layer Security) 프로토콜을 도입하여 이런 문제점을 보완함. TLS는 서버와 클라이언트를 오가는 모든 HTTP 메시지를 암호화함. 탈취당하더라도 이를 해석하는 것은 불가능하며 결과적으로 도청과 변조로부터 보호됨

## 키워드 ##
- ### HTTP ###
	- 웹 서버와 클라이언트가 리소스를 교환하기 위해 사용하는 프로토콜. 클라이언트가 요청하면, 서버가 응답하는 방식
- ### HTTP 메시지 ###
	- HTTP 서버와 클라이언트가 교환하는 데이터. 헤드와 바디로 구성되며, 각 줄은 CRLF로 구분됨
		- 헤드: 메시지에 대한 정보. 헤드의 끝에는 CRLF가 한 줄 있음
		- 바디: 클라이언트가 서버에게, 또는 서버가 클라이언트에게 전달할 데이터
- ### HTTP 요청 ###
	- 클라이언트가 서버에게 특정 동작을 요청하는 메시지
		- 메소드: 요청 URI가 가리키는 리소스에 대해, 서버가 수행했으면 하는 동작을 지정
		- 요청 URI: 메소드의 대상이 되는 리소스를 지정
- ### HTTP 요청 ###
	- 요청을 처리한 결과 및 이유,  그리고 클라이언트에 전송할 웹 리소스를 포함하는 메시지
		- 상태 코드: 요청을 처리한 결과
		- 처리 사유: 상태 코드가 발생한 이유
- ### HTTP 요청 ###
	- TLS를 이용하여 HTTP의 약점을 보완한 프로토콜
